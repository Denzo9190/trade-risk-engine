# ADR-018: Signal Processing Pipeline

**Date:** 2026-03-29  
**Status:** adopted

## Context
Strategies generate signals, but the system lacked a structured pipeline for signal validation and execution. Components were loosely coupled, making it hard to enforce consistent flow.

## Decision
Introduce **SignalProcessor** as the central component that processes `TradingSignal` objects generated by strategies.  
The pipeline becomes:
PriceUpdateEvent → StrategyEngine → TradingSignal → SignalProcessor → RiskEngine → ExecutionService


Benefits:
- Clear signal lifecycle
- Centralized validation and risk checks
- Clean separation of concerns
- Easy extension for multiple execution venues

## Consequences
- Strategies must implement the new `Strategy` interface returning `TradingSignal`.
- Existing `Signal` and `SignalExecutionService` may be deprecated in the future.
- All signals pass through a single processor, enabling logging, metrics, and audit.

## Alternatives Considered
- Continue using old `Signal` and `SignalExecutionService` – rejected because it lacks pipeline structure.
- Use message broker (Kafka) – overkill for current stage.
